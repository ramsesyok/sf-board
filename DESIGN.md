# サーバレス・チャット VSCode拡張 設計ドキュメント

## 1. 概要

オンプレミス・エアギャップ環境において、**サーバを一切立てずに** SMB/NFS 共有フォルダのみをトランスポート兼ストレージとして動作する、Slackライクなチャットツールを VSCode 拡張として実装する。

### 前提条件

| 項目 | 内容 |
|---|---|
| 環境 | エアギャップ(インターネット接続なし)、常駐サーバ不可 |
| データ共有手段 | 共有ネットワークフォルダ(SMB / NFS) |
| 想定同時利用規模 | 〜50人 |
| 必須機能 | チャンネル、スレッド返信、リアクション(絵文字)、ファイル添付 |
| メッセージ記法 | Markdown(生テキスト保存、クライアント側レンダリング) |
| リアルタイム性 | 準リアルタイム(反映遅延 数秒〜10秒程度を許容) |

### 設計原則(必ず守ること)

1. **追記専用(append-only)**: 共有フォルダ上のファイルは追記または新規作成のみ。既存内容の書き換え・削除は行わない。
2. **自分のファイルにしか書かない**: 各ユーザーは自分の userId を含むファイルにのみ書き込む。他ユーザーのファイルは読み取り専用。これによりロック機構なしで書き込み競合をゼロにする。
3. **状態ではなくイベントを保存**(イベントソーシング): 全操作をイベントとして操作者自身のログに追記し、各クライアントが全イベントをマージ・再生して現在状態を構築する。集計値・派生状態は共有フォルダに置かない。
4. **アトミック書き込み**: ファイル新規作成は「一時名で書き切る → リネーム」。JSONL追記は「1イベント=1行を単一 write で書く」。
5. **通知は冪等なトリガとしてのみ使う**: ファイル監視通知の内容(ファイル名・変更種別)を状態管理に使わない。通知を受けたら必ず実ファイルを読み直す。

## 2. 共有フォルダのディレクトリ構成

```
chat-root/                              # 共有フォルダのルート(設定で指定)
├── workspace.json                      # スキーマバージョン、ワークスペース名
├── users/
│   └── {userId}.json                   # プロフィール(displayName等)。本人のみ更新
├── cursors/
│   └── {userId}.json                   # 更新カーソル(§5)。本人のみ更新
├── channels/
│   └── {channelId}/
│       ├── channel.json                # チャンネル名、作成者、作成日時
│       └── events/
│           └── {YYYY-MM}/
│               └── {userId}.jsonl      # 当該ユーザーの当月イベントログ。本人のみ追記
└── attachments/
    └── {channelId}/
        └── {YYYY-MM}/
            └── {ulid}/
                ├── meta.json           # 元ファイル名、MIME、サイズ、SHA-256
                └── blob                # 実データ。書き込み後は不変(immutable)
```

- `channelId` はチャンネル作成時に生成した ULID。
- `userId` は OS ユーザー名等から取得し、設定で上書き可能とする。ファイル名に安全な文字(`[a-zA-Z0-9_-]`)に正規化する。
- 月別パーティション(`YYYY-MM`)により、過去月ファイルは不変とみなしてキャッシュ固定・再スキャン不要にする。

## 3. イベントスキーマ

イベントログは JSONL(1行1イベント、UTF-8、LF区切り)。

```typescript
type Ulid = string; // 26文字 Crockford Base32

type ChatEvent =
  | {
      id: Ulid;                    // イベントID。時系列順序・LWW判定の正はこれ
      type: "message_created";
      ts: string;                  // ISO 8601(表示用途のみ。順序判定に使わない)
      author: string;              // userId
      body: string;                // Markdown 生テキスト
      threadId?: Ulid;             // 親メッセージのイベントID(あればスレッド返信)
      attachments?: Ulid[];        // attachments/ 配下の ULID 参照
    }
  | {
      id: Ulid;
      type: "message_edited";
      ts: string;
      author: string;              // 元メッセージの author と一致する場合のみ有効
      targetId: Ulid;
      body: string;
    }
  | {
      id: Ulid;
      type: "message_deleted";
      ts: string;
      author: string;              // 元メッセージの author と一致する場合のみ有効
      targetId: Ulid;
    }
  | {
      id: Ulid;
      type: "reaction_added";
      ts: string;
      author: string;
      targetId: Ulid;
      emoji: string;               // 例 ":+1:" または Unicode絵文字
    }
  | {
      id: Ulid;
      type: "reaction_removed";
      ts: string;
      author: string;
      targetId: Ulid;
      emoji: string;
    };
```

### 状態の導出ルール(リデューサ仕様)

全チャンネル・全ユーザー・全月のイベントを **ULID 昇順** にマージソートして再生する。

- **メッセージ本文**: `message_created` を基点に、同一 `targetId` への `message_edited` のうち ULID 最大のものを現在本文とする。`message_deleted` があれば削除状態(UI上は「削除されました」表示)。編集・削除は `author` が元メッセージの `author` と一致するイベントのみ有効。不一致イベントは無視する。
- **リアクション**: 同一性は `(targetId, author, emoji)` の3つ組。同一3つ組に対する `reaction_added` / `reaction_removed` は **ULID 最大のイベントが勝つ**(last-write-wins)。結果が added なら表示。
- **スレッド**: `threadId` を持つメッセージを親 ID でグルーピング。返信数・最終返信時刻は読み取り時に導出する。共有フォルダに集計ファイルを置かない。
- 未知の `type` は無視して読み飛ばす(前方互換)。

### ULID 生成の単調性補正

時計ズレ対策として、生成した ULID が「自クライアントが最後に観測した ULID」以下になる場合は、観測済み最大値+1 に補正して単調性を保つ。表示順・LWW はすべて ULID 順を正とし、`ts` は表示専用とする。

## 4. 書き込みプロトコル

### 4.1 イベント追記(JSONL)

1. イベントを JSON 1行(末尾 LF)にシリアライズ。
2. 当月の自分のファイル `channels/{ch}/events/{YYYY-MM}/{me}.jsonl` に **単一の write** で追記し、flush する(Node: `fs.appendFileSync` 相当、可能なら fd を開いて `fsync`)。
3. 直後にカーソルファイル(§5)を更新する。

### 4.2 新規ファイル作成(channel.json / users / meta.json 等)

1. 同一ディレクトリ内に一時名 `{name}.tmp.{random}` で書き切る。
2. `fs.rename` で本来の名前へリネーム(同一ボリューム内なのでアトミック)。

### 4.3 添付ファイル

メッセージイベントが見えた時点で添付が必ず完成しているよう、順序を固定する:

1. `attachments/{ch}/{YYYY-MM}/{ulid}/` を作成し、`blob.tmp.{random}` に実データを書き切る。
2. SHA-256 を計算し、`meta.json` を書く(§4.2 の手順で)。
3. `blob.tmp.*` → `blob` にリネーム。
4. 最後に `attachments: [ulid]` を含む `message_created` イベントを追記。

読み手は `meta.json` の SHA-256 で `blob` の完全性を検証できる(検証はダウンロード保存時のみで可。表示のたびに行う必要はない)。

本文からの参照は独自スキーム `![alt](attachment://{ulid})` を用い、レンダラ側で実パス(Webview の場合は `asWebviewUri` 経由)に解決する。

### 4.4 カーソルファイル更新

イベント書き込み後、`cursors/{me}.json` を §4.2 の手順(tmp→rename)で丸ごと書き換える:

```json
{
  "lastEventId": "01J8XK...",
  "lastChannelId": "01J7AA...",
  "updatedAt": "2026-07-08T10:23:45.120+09:00"
}
```

## 5. 更新検知(監視+照合ポーリングのハイブリッド)

新着検知の対象は `cursors/` ディレクトリ **1つだけ** に絞る。チャンネルごとのイベントディレクトリを個別監視してはならない(監視ハンドル増加は障害点増加)。

### 5.1 三層構成

| 層 | 手段 | 周期/契機 | 役割 |
|---|---|---|---|
| 監視層 | `fs.watch("chat-root/cursors", { recursive: false })` | プッシュ通知 | 待機中アクセスほぼゼロで新着を即検知(SMB の CHANGE_NOTIFY を利用) |
| 照合層 | `readdir` + 各カーソルの mtime/size 比較 | 60〜120秒 + ジッタ | 監視の取りこぼし検出の安全網 |
| フォールバック層 | 適応ポーリング | 2〜5秒 + ±30% ジッタ | 監視が信頼できない環境(NFS等)用 |

### 5.2 動作フロー

1. `fs.watch` の通知は **300ms デバウンス** し、通知内容は無視して `cursors/` を読み直す。
2. 前回値から変化したカーソルを検出したら、該当ユーザーのカーソルを読み、`lastChannelId` のイベントファイルを差分読みする(§5.3)。
3. カーソルは全チャンネル横断の「最後の1件」しか示さないため、変化検出時はそのユーザーの当月イベントファイル群のサイズも確認し、変化のあったものすべてを差分読みする。

### 5.3 差分読み(オフセット管理)

- 各 `{userId}.jsonl` について「前回読了バイトオフセット」をローカルに保持し、追記分のみ読む。
- 読み取った末尾が LF で終わらない場合、その不完全行は捨てて次回に持ち越す(書き込みフラッシュ途中の読み取り対策)。
- SMB クライアントキャッシュ対策として、ポーリング/照合時はファイルハンドルを使い回さず毎回開き直す。

### 5.4 自動フォールバック(ヘルスチェック)

- 「照合ポーリングが変化を発見したが、直近(照合周期内)に監視層から通知が来ていなかった」回数をカウント。
- 閾値(例: 連続3回)を超えたら監視層を諦め、フォールバック層(適応ポーリング)へ切り替える。
- `fs.watch` の `error` イベント発生時、およびネットワーク切断からの復帰検出時は監視ハンドルを張り直す。復旧確認後は監視層に戻ってよい。

### 5.5 適応ポーリング(フォールバック層の仕様)

- 基本周期に ±30% の乱数ジッタを加え、機械的周期性を崩す。
- VSCode ウィンドウ非アクティブ時は 30〜60秒に延伸。直近に新着があった場合のみ短周期化。
- 開いているチャンネルは短周期、それ以外はカーソル経由の検知のみとする段階制御。

> 補足: chokidar はネットワークドライブでポーリングにフォールバックするため使用しない。VSCode の `FileSystemWatcher` もネットワークドライブでの信頼性が低いため使用しない。素の `fs.watch` を直接使う。

### 5.6 同期診断ログ(任意・既定で無効)

本層(監視・照合・フォールバック・ヘルスチェック)の挙動は、任意の診断ログで観測できる。既定では無効で、設定で有効にしたときのみローカルに記録する(実環境の通知到達・モード切替の調査用)。出力先・記録内容・実装は `DESIGN_EXTENSION.md` §8.1 を正とする。記録するのはメタデータ(カーソル名・件数・ULID・モード等)のみで、メッセージ本文は記録しない。外部送信は行わず、共有フォルダにも書き込まない(エアギャップ原則を厳守)。

## 6. クライアントローカル状態(共有フォルダに置かないもの)

保存先: `ExtensionContext.globalStorageUri` 配下。

- **既読位置**: チャンネルごとの最終読了 ULID。未読バッジはローカル計算。
- **読み込みオフセット**: 各イベントファイルの読了バイト位置。
- **マテリアライズ済み状態キャッシュ**: リデューサ適用後のメッセージ状態。過去月分は不変としてキャッシュ固定。起動時は前回オフセットからの差分再生のみ行う。
- キャッシュはスキーマバージョンを持ち、不整合時は全再生で再構築できること(キャッシュ破棄=フルスキャンが常に安全な復旧手段)。

## 7. VSCode 拡張の構成

```
src/
├── extension.ts            # activate/deactivate、設定読み込み
├── core/
│   ├── ulid.ts             # ULID 生成(単調性補正付き)
│   ├── events.ts           # イベント型、シリアライズ/パース(不正行はスキップ)
│   ├── reducer.ts          # イベント列 → チャンネル状態(§3 導出ルール)
│   ├── store.ts            # 共有フォルダ読み書き(§4 プロトコル)
│   ├── sync.ts             # 監視+照合+フォールバック(§5)
│   └── localCache.ts       # §6 ローカル状態
├── ui/
│   ├── channelTree.ts      # TreeView: チャンネル一覧+未読バッジ
│   ├── chatPanel.ts        # Webview: 会話表示・入力
│   └── webview/            # Webview 側 HTML/JS/CSS
└── test/
```

### UI 要件

- サイドバーに TreeView(チャンネル一覧、未読数バッジ、チャンネル作成コマンド)。
- チャンネル選択で Webview パネルを開く。メッセージ一覧(スレッドは親の下に折りたたみ)、リアクションのトグル UI、添付のアップロード/ダウンロード、Markdown 入力欄(Ctrl+Enter 送信)。
- 新着は差分レンダリング(全再描画しない)。

### Markdown レンダリング

- `markdown-it` を Webview 側で使用。`html: false` とし、生 HTML は無効化。
- 出力は DOMPurify 等でサニタイズ。リンクは `target` 制御し、外部プロトコルは無効化(エアギャップ前提)。
- `attachment://` スキームの画像参照を Webview URI に解決するプラグインを実装。
- コードブロックのシンタックスハイライトは任意(highlight.js をバンドル可)。

### 設定項目(contributes.configuration)

| キー | 内容 | 既定値 |
|---|---|---|
| `chat.rootPath` | 共有フォルダの chat-root パス | (必須) |
| `chat.userId` | ユーザーID(未設定時は OS ユーザー名) | "" |
| `chat.displayName` | 表示名 | "" |
| `chat.poll.reconcileSec` | 照合周期 | 90 |
| `chat.poll.fallbackSec` | フォールバック基本周期 | 4 |
| `chat.watch.enabled` | 監視層の有効化 | true |

## 8. エラー処理・堅牢性

- JSONL パース失敗行は警告ログを出してスキップ(1行の破損で全体を止めない)。
- 共有フォルダ到達不能時: UI に「オフライン」表示。送信操作はローカルの送信キューに積み、復帰後に追記する(キューもローカル保存し、拡張再起動で消えないこと)。
- workspace.json の `schemaVersion` が未知の場合は読み取り専用モードで起動し、更新を促す。
- 同一ユーザーが複数マシンで同時起動した場合も、書き込みは追記のみなので破損はしない(同一ファイルへの同時追記はまれに行が交錯し得るため、userId にホスト名サフィックスを付ける設定を用意してもよい)。

---

## 9. 実装フェーズ提案

1. **Phase 1**: core 層(ulid / events / reducer / store)+ ユニットテスト。ローカルフォルダで動作確認。
2. **Phase 2**: sync 層(照合ポーリングのみ)+ TreeView + Webview の最小チャット(投稿・表示・Markdown)。
3. **Phase 3**: スレッド、リアクション、添付。
4. **Phase 4**: fs.watch 監視層+ヘルスチェック付きフォールバック、送信キュー、未読管理、パフォーマンス調整。

## 10. 非機能・運用メモ

- アクセスパターン試算(50人): ハイブリッド構成のアイドル時は照合 40 op/時/人 程度+待機中の CHANGE_NOTIFY のみ。ファイルサーバの通常業務負荷に埋もれる水準。
- 全データが平文 JSONL のため監査可能。導入前にセキュリティ部門へ本ドキュメントのアクセスパターン見積もりを添えて事前承認を得ること。
- アクセス制御は共有フォルダの ACL に委譲する(拡張側で認証は実装しない)。
- 同期診断ログ(§5.6 / `DESIGN_EXTENSION.md` §8.1)は既定で無効。トラブルシュート時のみ有効化し、ローカルの出力チャネルにメタデータのみを記録する(本文は残さない・外部送信なし)。
